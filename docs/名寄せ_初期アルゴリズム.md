# 名寄せ：初期の現実解（MVP で必須）

最強版の完全自動名寄せは後回しでよい。**MVP 時点で「同一っぽい候補を出し、人が確定する」** までを決めておかないと運用が崩れる。ここではそのルールを明文化する。

---

## 1. 方針

- **100% 自動統合はしない**。候補を提示し、**人が「同一」を確定**する。
- 確定後は、既存の `Talent` / `Project` に `TalentOffer` / `ProjectOffer` を紐づける（または新規 Talent/Project を作成して紐づける）。

---

## 2. 人材（Talent）の同一判定キー（初期版）

次の組み合わせで「同一人物の候補」を出す。**すべて一致しなくてもよい**（スコア化して上位を候補表示する想定）。

| 項目 | 扱い |
|------|------|
| 氏名 | 正規化（スペース除去・カナ統一）して比較。**伏字（○田、山○等）あり想定**で、部分一致も候補に含める。 |
| 年齢 | 完全一致 or ±1 歳を同一候補とする。 |
| 最寄駅 | 正規化後の `station_id` 一致。 |
| 主要スキル上位3 | スキルIDの集合で比較（順不同でよい）。2/3 以上一致で候補。 |

**候補スコアの例**

- 氏名一致＋年齢＋最寄＋スキル3: 強候補
- 氏名部分一致＋年齢＋スキル2: 候補
- 下記「強い証拠」があればスコア加算

---

## 3. 強い証拠：添付スキルシートのハッシュ

- **同一ファイル**（同じバイナリ）が添付されていれば、**同一人物の強い証拠**とする。
- 取り込み時に添付ファイルの SHA-256 ハッシュを計算し、`raw_attachments` に `content_hash` として保存する（スキーマに追加推奨）。
- 人材メール同士で `content_hash` が一致すれば「同一人物候補」をトップに表示する。

---

## 4. 案件（Project）の同一判定キー（初期版）

| 項目 | 扱い |
|------|------|
| クライアント名・案件名 | 正規化して比較（伏字・略称あり想定）。 |
| 必須スキル上位3 | スキルIDの集合で比較。 |
| 単価帯 | 同一帯（例：70〜80万）で候補。 |
| 入場時期 | 近い時期（例：同月）で候補。 |

案件も **「候補提示 → 人が確定」** とする。

---

## 5. UI/運用

- **一覧**: 新規取り込みの TalentOffer / ProjectOffer について、上記キーで「同一候補」を検索し、**「同じ人/同じ案件の可能性」として一覧表示**する。
- **確定**: ユーザーが「同一」を選択 → 既存 Talent/Project に Offer を紐づけ、または新規 Talent/Project を作成して紐づける。
- **ログ**: 誰がいつどの Offer をどの Talent/Project にマージしたかを `activities` に記録する（action: `merge_talent` / `merge_project`）。

---

## 6. スキーマで持つとよいもの（MVP）

- `raw_attachments.content_hash`（任意）: 添付の同一性判定用。
- 名寄せ候補の計算結果は **都度計算** でもよい（キャッシュする場合は `merge_candidates` のような一時テーブル or ビューでよい）。

この「最初の現実解」を実装しておけば、MVP でも二重提案を減らしつつ、最強版でアルゴリズムを強化できる。
