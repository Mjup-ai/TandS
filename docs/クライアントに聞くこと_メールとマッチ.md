# クライアントに聞くこと（メール）＆ 受信後のマッチの流れ

このシステムを動かすために**メールで何を聞くか**と、**受信できた後のマッチをどうするか**をまとめる。

---

## 届くメールは2種類

- **人材情報が届くメール** … 要員提案・スキル・希望単価・入場時期・スキルシート参照などが書かれたメール。システムでは「人材」に分類し、人材台帳（TalentOffer）に載せる。
- **案件情報が届くメール** … 案件概要・必要スキル・単価・稼働場所・入場時期などが書かれたメール。システムでは「案件」に分類し、案件台帳（ProjectOffer）に載せる。

この2種類が届く先は、**1アカウントに集約されている場合も、アカウントが2つ（人材用・案件用など）に分かれている場合**もある。

---

## アカウントが2つある場合

- 例：**人材情報用メールアドレス** と **案件情報用メールアドレス** が別々にある。
- システム側では **2アカウントとも連携**し、どちらから届いたメールも同じ「受信一覧」に取り込む。取り込み後に「人材」「案件」で分類する（アカウントで判別できる場合は、取り込み時から分類を付与することも可能）。
- Gmail API なら **アカウントごとに OAuth 認証** を1回ずつ行い、2つ分のメールを取得する実装にする。

---

## 1. クライアントのメールで聞くこと（これが分かればシステムは動く）

**自社の前提**: さくらインターネットの Web メールを使用しており、受信後に任意のアドレスへ転送している。  
→ 取り込み対象は **転送先のメールボックス**。転送先がどこか（Gmail / さくら / 他）で Gmail API か IMAP かが決まる。

自動受信（API連携）をするために、次のことを聞く。

| 聞くこと | 聞く理由 | 例 |
|----------|----------|-----|
| **メールはどこで受けているか（転送先はどこか）** | Gmail / Outlook / さくら 等で取り込み方式が変わる | 「さくらから Gmail に転送している」「転送先もさくらの Web メール」など |
| **独自ドメイン（@自社ドメイン）を使っているか。使っている場合、そのメールの「預かり先」はどこか** | アドレスが @company.co.jp でも、**預かり先**が Google → Gmail API、Microsoft/自社/さくら → Graph や IMAP。 | 「独自ドメインで Google Workspace」「さくら Web メールで、転送先は Gmail」など |

### 相手に送る質問の例（預かり先を聞く）

転送先や集約先のアドレス（例: **ses@vi-hack.com**）が分かっている場合、次のように聞くと答えやすい。

- **「そのメール（転送先 ses@○○.com）は、どこで受信されていますか？ Google（Google Workspace） / Microsoft 365 / 自社サーバー のどれに該当しますか？」**
- または: **「ses@○○.com のメールは、Gmail 宛に届いていますか？ それとも別のサービス（Outlook 等）で受けていますか？」**

答えが「Google Workspace」「Gmail で受けている」→ Gmail API で取り込み可能。  
「Microsoft 365」「自社サーバー」「IMAP で取れる」→ IMAP または Graph で取り込み。
| **アカウントは1つか、2つ（以上）か。2つの場合、人材用・案件用など役割の区別があるか** | 2アカウントなら両方に連携する。役割が分かれていれば、取り込み時に分類を自動付けられる | 「1アカウント」「2アカウント（人材用と案件用）」 |
| **各アカウントに外部アプリのアクセスを許可できるか** | Gmail API ならアカウントごとに「メール読み取り」許可が必要 | 「2つとも Google で、API 利用OK」 |
| **メールの保管・削除ルール** | 読んだメールを「既読のまま残す」か「別ラベルに移す」か。システムが「未読」や「特定ラベル」で取得する前提を決める | 「未読で取る」「ラベル『SES案件』をつけたものだけ取る」 |
| **添付（スキルシート）は必須で見るか、本文だけでよいか** | 添付まで解析するかで工数・コストが変わる。最初は本文だけでも動く | 「まずは本文だけでよい」「スキルシートPDFも読みたい」 |

**これが決まれば**  
→ Gmail なら Gmail API（OAuth）で「そのアカウントの未読 or 指定ラベル」を定期取得する実装にできる。Outlook/社内なら IMAP や Microsoft Graph の検討になる。

## 2. 受信できた後のマッチはどうしたらいいか（流れ）

受信までできたら、あとは次の流れで「マッチ」を使う。

| ステップ | やること | システム側 |
|----------|----------|------------|
| **1. 分類** | 届いたメールを「案件」か「人材」に振り分ける | 受信一覧で各行の「人材」「案件」ボタンで分類。将来はAIで自動分類も可 |
| **2. 台帳に出る** | 分類したものが案件タブ・人材タブに並ぶ | 自動で ProjectOffer / TalentOffer に登録される |
| **3. マッチを見る** | マッチタブで「案件×人材」の候補とスコア・推薦が出る | 推薦だけフィルタで絞れる。スコア内訳・除外理由も表示 |
| **4. 人が連絡する** | おすすめ候補に対して、**営業がメール or 電話で提案・確認**する | システムは自動送信しない。テンプレ文や電話メモは将来生成できる |
| **5. 結果を記録する（任意）** | 提案した結果「採用した」「見送り（理由あり）」を記録する | 評価ループ用。重み調整や分析に使える |

**一言でいうと**  
「受信 → 分類 → 案件・人材タブで確認 → マッチタブでおすすめを見る → **人が連絡** → 必要なら結果を記録」までやれば、マッチは一通り回る。

---

## 3. まとめ

- **メールで聞くこと**: どこで受けているか・1つに集まっているか・API/アクセス許可・未読/ラベル・添付をどうするか。**これが分かれば自動受信の実装ができる。**
- **マッチのやり方**: 分類 → 台帳で確認 → マッチタブで候補を見る → **人が連絡** → 任意で結果を記録。システムは「候補と理由を出す」までで、送信は人が行う。
