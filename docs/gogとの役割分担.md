# gog（gogcli）との役割分担

gog は「Google Workspace を CLI 経由で操作し、AI に指示してやらせる」ツール。  
参考: [「gog」導入・設定ガイド](https://zenn.dev/takna/articles/gog-cli-setup-guide)

**結論**: gog **だけ**でこのシステムは完結しない。一方で **「Google 側の入出力」** は gog でかなり賄える。  
→ **gog ＝ 入出力の手足**、**中核ロジック（名寄せ・マッチ・DB）＝ 別プロセス（このアプリ）** の2層構成が現実的。

---

## gog でできること（このシステムで使える部分）

| 役割 | 内容 |
|------|------|
| **Gmail** | 転送集約されたメールの検索・取得（本文・ヘッダ）。専用アカウント2つ分を CLI で取得するのも可能。 |
| **添付** | スキルシート等の添付を取得してローカル保存 or Drive へ。 |
| **Sheets** | 抽出結果や台帳を Google スプレッドシートに書き込む。**スプシ完結 MVP の UI** にできる。 |
| **通知・リマインド** | Tasks / Calendar に載せる。 |

→ セットアップは「Google Cloud プロジェクト・API 有効化・OAuth 同意・credentials.json → `gog auth add`」で、**Gmail API を自前実装するより手軽**な場合がある。

---

## gog だけでは足りないこと（この設計の“心臓”）

| 役割 | 内容 |
|------|------|
| **名寄せ** | 同一人材・同一案件の統合。オファー概念での商流管理。 |
| **マッチング** | Hard Filter → スコアリング → 理由生成 → マッチ履歴の **DB 管理**。 |
| **運用・防止** | 500件/日の運用ログ、重複提案防止、評価ループ。 |

これらは **DB とアプリ側のロジック** でやる想定。gog は「メールを取ってくる」「シートに書く」までの入出力に使う。

---

## 現実的な組み合わせ

- **スプシ完結 MVP（提案①）**  
  **gog** で Gmail → メール取得 → 分類・抽出結果を **Sheets に書き込み** までやる。マッチはシート上で簡易スコア or 手動。名寄せは「候補提示」程度ならシートでも可能。
- **専用システム版（提案②）**  
  **gog** は「Gmail から未読を取ってきて、中身をこのアプリに渡す」**入出力**に使う。名寄せ・マッチ・履歴は **このアプリ（DB）** で完結。  
  または、Gmail 取り込みは **このアプリで Gmail API を直接叩く**実装にして、gog は使わない選択もある。

**一言**: gog でこの仕組みを**完結**は無理。**「Google 側（メール取得・添付・シート書き込み）」は gog で賄い、「名寄せ・マッチ・DB」はこのシステム**でやる、という分担にするとよい。

---

## 今設計しているシステムで「gog は使えそう？」の答え

| どれを指すか | 使えそう？ | 理由 |
|--------------|------------|------|
| **提案① スプシ版** | **◎ かなり使える** | Gmail 取得 → 分類・抽出の結果を **Sheets に書き込む**までを gog でやれる。スプシが UI なので gog と相性がいい。Apps Script の代わりや、手動 .eml 投入の代わりになる。 |
| **提案② 専用システム（Webアプリ）** | **△ 補助なら可、メインには向かない** | 専用システムは「バックエンドが Gmail API を直接叩いて DB に保存」の方が自然。gog を使うなら「gog でメール取得 → ファイル or キューに吐く → バックエンドが読みにいく」のような連携になり、構成がやや複雑になる。**Gmail API をバックエンドに実装する方がシンプル**。 |

**まとめ**: **スプシ版（①）なら gog は十分使える。専用システム（②）なら、メール取り込みはバックエンドで Gmail API を叩く実装を推奨。** gog は「入出力の手足」として使うなら、どちらでも選択肢にはなる。
