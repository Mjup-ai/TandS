# 1. 業務フロー（現状 → 新運用）

## 前提：届くメールは2種類

- **人材情報が届くメール** → 人材台帳に載せる
- **案件情報が届くメール** → 案件台帳に載せる

この2種類が受信トレイに届き、転送で集約されている前提で以下を考える。

## 現状の課題

- 毎日500件級のメールが4名に分散
- 転送で集約しているが「探す」が手作業
- 同一人材/同一案件の重複提案・二重評価・商流の深さによる条件悪化
- 「なぜこの候補か」の説明が属人化

## 新運用の流れ

```
[4名のメール] → 転送で集約 → 専用受信アカウント
       ↓
   Ingest（原本保存）
       ↓
   分類（案件/人材/混在/その他）
       ↓
   抽出（固定JSON + confidence + 根拠行）
       ↓
   正規化（スキルID・駅・単価・雇用形態）
       ↓
   名寄せ（同一人物・同一案件の統合）
       ↓
   台帳（TALENTS / PROJECTS）更新
       ↓
   検索（プリセット・ファセット・自然文）
       ↓
   マッチ（Hard Filter → スコア → 70点以上を推薦）
       ↓
   表示：推薦理由・注意点・確認質問・テンプレ文
       ↓
   人が連絡（メール/電話）※自動送信なし
```

## 役割分担

| 処理 | 担当 | 備考 |
|------|------|------|
| メール転送・集約 | 運用（現状どおり） | 4名→1アカウント |
| 受信・保存・分類・抽出 | システム | イベント駆動（メール着信時） |
| 名寄せ確定・要確認項目の修正 | 人 | 候補はシステム提示 |
| 検索・マッチ結果の確認 | 営業 | プリセット・ファセット |
| 提案・面談・成約の連絡 | 人 | テンプレ/メモはシステム出力 |

## 成功指標（業務側）

- 1件の案件に対する**初回候補提示まで**の時間を現状の約**1/10**に短縮
- 「探す」時間削減により**提案数**を増やす
- 候補に対して「スコア内訳・根拠行」で説明可能にし、納得感と次の行動を支援
