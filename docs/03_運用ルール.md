# 3. 運用ルール（ガードレール）

## 必須運用ルール

### 連絡は人が行う

- **自動送信しない**。提案メール・リマインド・成約通知など、一切の自動送信を行わない。
- システムは「下書き・テンプレ・電話メモ」を生成し、送信・架電は必ず人が実施する。

### PII・権限

- 個人情報（氏名・連絡先・スキルシート）は権限で制御する。
- 閲覧者ロールではマスキングまたは制限付き表示とする方針を推奨。

### 原本保全

- メール原文・添付・URLは削除せず保持する（監査・再解析用）。
- `raw_emails`, `raw_attachments`, `raw_links`, `ingest_events` は論理削除のみとし、物理削除は行わない。

### 抽出値の信頼性

- 抽出した「単価」「年齢」等には**根拠行**を紐づけ表示する。
- confidence が低い項目は「要確認」に回し、全件チェックは強制しない（500件/日対応のため）。

### コスト上限

- AI利用（分類・抽出・正規化・マッチ理由生成）について、**1日上限・月上限**を設ける。
- 想定を超えた場合は停止または保留とし、アラートを出す。

## 承認・ログ

- 名寄せの確定（同一人物/同一案件のマージ）は**手動承認**を推奨。
- 閲覧・確定・提案・成約などの重要操作は `activities` に記録する（誰が・いつ・何をしたか）。

## 運用事故防止のまとめ

| リスク | 対策 |
|--------|------|
| 二重提案 | 名寄せ＋同一人物のオファー束ねで最良条件を表示、重複提案フラグ |
| 抽出ミス | 根拠行表示＋要確認運用＋推測禁止（欠損はnull） |
| 自動送信事故 | 自動送信機能を実装しない |
| コスト暴走 | 日/月上限・アラート・保留 |
| 監査不足 | 原本保全＋ingest_events＋activities ログ |

以上を「運用ルール」として開発・運用側で共有する。
