# 取り込み仕様（Ingest）・原本保全の定義

転送メールの形式（ヘッダが潰れる／引用が付く／添付が消える等）で難易度が変わる。**取り込み方式を1つ決め、「原本保全」を1枚で定義する**。

---

## 0. 自社メール環境（前提）

- **自社メールサーバー**: さくらインターネットの Web メールを使用している。
- **運用**: メール受信後に**任意のアドレスへ転送**する設定をしている。

この場合、システムで取り込む対象は **転送先のメールボックス** になる。

| 転送先 | 取り込み方式 |
|--------|----------------|
| **Gmail / Google Workspace** | Gmail API（OAuth）で転送先アカウントのメールを取得 |
| **さくらのまま（転送先もさくら）** | さくらが提供する IMAP/POP で転送先メールボックスに接続（要確認） |
| **その他（Outlook 等）** | IMAP または該当サービスの API |

**次の確認事項**: 転送先のアドレスが「どこで受信されているか」（Gmail / さくら / 他）を決めれば、上表のどれで取り込むかが決まる。クライアントへの質問は `docs/クライアントに聞くこと_メールとマッチ.md` を参照。

---

## 1. 取り込み方式の選択

| 方式 | メリット | デメリット | 推奨タイミング |
|------|----------|------------|----------------|
| **手動 .eml 投入** | 実装が不要、ヘッダ・添付をそのまま保持できる | 運用負荷、自動化なし | **MVP最初期**（抽出・DB化を先に固める） |
| **Gmail API** | ラベル・スレッド・添付を制御しやすい、転送先1アカウントで取得可能 | OAuth・スコープ・制限の理解が必要 | 取り込み自動化を入れる段階 |
| **IMAP** | 汎用、多くのメール環境で使える | 転送時のヘッダ改変・引用の扱いが環境依存 | Gmail以外が前提のとき |
| **転送ルール（クライアント側）** | ユーザー操作のみ | 転送先で「元のFrom/Message-ID」が残るかは環境次第 | 検証必須 |

**決定ルール**

- **MVP（2週間で動く範囲）**: まず **「手動で .eml 投入」** に固定する。  
  → 入力源は後から差し替え可能なので、**抽出とDB化を先に固める**。
- **本番運用**: 集約方法（4名→1アカウント）が決まったら、**Gmail API または IMAP** のどちらで取り込むかを決め、本ドキュメントに追記する。

---

## 2. 転送メールで起きがちなこと（検証メモ）

- **ヘッダが潰れる**: 転送元によっては `Message-ID` / `References` が落ちる・変わる。
- **引用が付く**: 「転送メール」として本文の前に引用ブロックが付く。Normalize で除去対象。
- **添付が消える**: 転送ルールやサイズ制限で添付が落ちる場合あり。→ 添付必須の案件は別ルート（API直接取得等）を検討。
- **From/To**: 転送先では「転送元」が From になるか、「元の送信者」が別ヘッダ（X-Forwarded-From 等）に残るかは環境依存。**どのヘッダを「発信者」とするか**を取り込み仕様で固定する。

---

## 3. 原本保全の定義（1枚）

**「原本」として保持するもの**

| 対象 | 内容 | 保存場所（例） |
|------|------|----------------|
| メール本文 | 生の text/plain および text/html（取得できる場合） | DB の `raw_emails.body_text` / `body_html` またはオブジェクトストレージ参照 |
| メールメタ | Message-ID, 受信日時, From, To, Subject, その他必要なヘッダ | `raw_emails` のカラム |
| 添付 | ファイル実体 | オブジェクトストレージ（S3相当/Drive）＋ `raw_attachments.storage_key` で参照 |
| 本文内URL | スキルシート等のURL | `raw_links` |

**扱いの原則**

- **論理削除のみ**: 原本は物理削除しない。`deleted_at` で無効化。
- **上書きしない**: 同一 Message-ID の再取り込みは「スキップ」または「新バージョンとして別レコード」とし、既存本文を上書きしない。
- **監査**: 誰がいつ取り込んだかは `ingest_events`（eventType: `ingested`）に記録する。

**添付の保管場所**

- MVP: ローカル or プロジェクト内ストレージ（パスを `raw_attachments.storage_key` に格納）。
- 本番: S3相当 or Google Drive 等を決め、**データ保持と権限**（別 doc）の「保存期間」「アクセス権」に従う。

---

## 4. MVP 時の実装方針（最短）

1. **入力**: 手動で .eml ファイルを指定ディレクトリに置く（またはアップロードAPI 1本）。
2. **パース**: .eml からヘッダ・本文・添付を分解し、`raw_emails` / `raw_attachments` / `raw_links` に格納。
3. **原本はすべてDB（＋ファイル実体）に保存**し、上書きしない。
4. Gmail/IMAP 連携は **「取り込み仕様」を本ドキュメントに追記したうえで** 後から実装する。

これで「取り込み仕様の確定」と「原本保全の定義」を1枚に揃え、実装時の詰まりを防ぐ。
