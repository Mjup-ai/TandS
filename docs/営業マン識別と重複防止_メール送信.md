# 営業マン識別・重複防止・マッチからメール送信

営業マンが6名おり、各メールが1つのGmailに転送されて集まる前提で、**どの営業マンから届いたメールか**を分かるようにし、**同一営業先への重複営業を防ぎ**、**マッチ画面からメールを送れる**ようにする設計。

---

## 1. 営業マン識別（6アカウント → 1Gmail に集約の場合）

### 課題

6人のメールアカウントが**1つのGmailに転送**されると、転送先の受信トレイには「誰の元メールか」がヘッダに残っていない場合がある。

### 方式の比較

| 方式 | どの営業マンか | 実装 | 推奨 |
|------|----------------|------|------|
| **A. 6アカウントを個別連携** | 取得元アカウント = 営業マン | Gmail API で6アカウントそれぞれ OAuth し、取り込み時に「このメールはアカウントXで取得」と記録 | ◎ 確実 |
| **B. 1Gmail のみ連携（転送で集約）** | 転送元をヘッダから推定 | 転送時に付く `X-Forwarded-From` / `Delivered-To` / 本文の「転送元」表記をパース | △ 環境依存・要検証 |
| **C. 転送時にラベル付け** | ラベル = 営業マン | 各営業マンの転送ルールで「ラベル: 営業マンA」などを付与し、Gmail API でラベルを取得 | ○ 運用で解決 |

### 推奨: A（6アカウント個別連携）

- 取り込み元を「メールアカウント（= 営業マン）」と1:1で紐付ける。
- DB: `RawEmail` に **`receivedByAccountId`**（または `salesPersonId`）を追加し、取り込み時に「どのアカウントで取得したか」を保存。
- マスタ: **メールアカウント（営業マン）** テーブル（id, 表示名, メールアドレス, OAuth 用識別子 など）を用意する。

**1Gmail に転送する運用のままにする場合**

- **B**: さくら→Gmail 転送で、転送元アドレスがヘッダに残るか検証する。残るならそのヘッダをパースして「転送元 = 営業マン」のマスタと照合。
- **C**: Gmail の転送ルールで「転送元アドレスごとにラベルを付ける」ようにし、API 取得時にラベルから営業マンを判定する。

## 2. 重複防止（同一営業先への二重営業）

### 営業先の定義

- **案件**: 発信元（From）や本文・件名から分かる「クライアント企業・担当者」。
- **人材**: 発信元（From）や本文から分かる「人材・BP 名」。
- 重複とは「**同じ営業先（同じクライアント or 同じ人材）に、別の営業マンが別メールで営業している**」状態。

### 重複検知に使うキー

| 種別 | キー候補 | 備考 |
|------|----------|------|
| 案件 | 発信元ドメイン / 正規化した From / 企業名（抽出） | 同一企業に複数営業マンが連絡しているか |
| 人材 | 発信元メールアドレス / 正規化した From / 氏名（抽出） | 同一人材に複数営業マンが連絡しているか |

### 実装方針

1. **RawEmail に営業マンを持たせる**（上記 `receivedByAccountId`）。
2. **営業先の正規化**: 案件は `From` のドメイン or 企業名、人材は `From` アドレス or 氏名を正規化して保持（既存の Partner / Company や新規の `SalesTarget` など）。
3. **一覧・マッチ画面で「同一営業先」を表示**:  
   同じ営業先に別の営業マンが紐づくメールがある場合、「他に ○○ さんがこの営業先に連絡済み」と表示する。
4. **重複防止の運用**: 送信前に「この営業先には既に 営業マンB がメール送っています」と警告し、二重送信を避ける。

まずは **「どのメールがどの営業マンか」を分かるようにする** ことが前提。その上で **営業先キーを決め、同一キーのメールを一覧で見せる** ところから実装する。

## 3. マッチからメールを送る

### 要件

- マッチした内容（案件×人材）を見られる。
- 案件メール・人材メールを押すと、**こちらからメールを送れる**（営業マンの送信操作を支援）。

### 送信方式の選択

| 方式 | 内容 | メリット | デメリット |
|------|------|----------|------------|
| **mailto リンク** | 案件/人材の「送信元アドレス」を To にした mailto を開く。件名・本文テンプレは任意。 | 実装が簡単、追加認証不要。既存のメールクライアント（Gmail 等）で送信。 | 送信はユーザーの環境に依存。 |
| **Gmail API で送信** | システムから Gmail API で送信。 | 送信実績をシステムに残せる。 | 各営業マンアカウントの OAuth と「誰のアカウントで送るか」の選択が必要。 |

**MVP**: **mailto リンク** で「案件の送信元に返信」「人材の送信元に返信」を開く。  
マッチ画面で「案件メール」「人材メール」を押すと、それぞれのメールの **From** を宛先にした mailto が開く。件名・本文は空または簡単なテンプレでよい。

### UI

- マッチ1件ごとに「案件」「人材」のブロックを表示。
- 各ブロックに **「この案件にメールを送る」** / **「この人材にメールを送る」** ボタン（リンク）を配置。
- クリックで `mailto:案件のfromAddr` または `mailto:人材のfromAddr` を開く。必要なら `subject=` `body=` でテンプレを付与。

---

## 4. まとめ

| 項目 | やること |
|------|----------|
| **営業マン識別** | 6アカウントを個別連携し、取り込み時に「取得元アカウント」を DB に保存。1Gmail 集約の場合は転送元ヘッダ or ラベルで推定。 |
| **重複防止** | 営業先キー（案件=発信元/企業、人材=発信元）を決め、同一キーに他営業マンが紐づく場合は一覧・マッチで表示・警告。 |
| **マッチから送信** | マッチ内容を表示し、案件・人材それぞれ「メールを送る」で mailto を開く（宛先 = 各メールの From）。 |

実装順: まず **マッチ画面で案件・人材の fromAddr を表示し、mailto で送信** まで実装。その後、**RawEmail に receivedByAccountId と営業先キーを追加** し、営業マン表示・重複表示を追加する。
