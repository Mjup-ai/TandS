# 評価ループ（70点の調整方法）

「良い/悪い」のフィードバックをどこに記録するかが重要。**matches に「採用/見送り/理由」を残し、重み調整に使う**。

---

## 1. 方針

- マッチ候補に対して、営業が **採用した / 見送った** を記録する。
- **見送り理由**（任意だが推奨）を選べる or 自由文で残す。
- このデータを **スコア重み・Hard Filter ルールの見直し** に使う（評価ループ）。

---

## 2. 記録する項目（matches に持つ）

| 項目 | 型 | 説明 |
|------|-----|------|
| outcome | enum | `pending` / `accepted` / `rejected` |
| outcome_reason | string | 見送り理由コード or 短い理由（例: "単価折り合わず", "入場遅い"） |
| outcome_notes | string | 自由メモ（分析用） |
| outcome_at | datetime | 採用/見送りを記録した日時 |
| outcome_by | string | 記録したユーザーID |

※ スキーマでは `Match` にこれらのフィールドを追加する（後述）。

---

## 3. 見送り理由の選択肢（例）

- 単価が合わない  
- 入場時期が合わない  
- スキル不足  
- 勤務形態が合わない  
- その他（outcome_notes に記載）

選択肢は CONFIG またはマスタで管理し、分析しやすくする。

---

## 4. 重み調整への使い方

- **accepted** の割合が高い条件（例: 必須スキル一致が高いスコアの候補）→ その項目の重みを維持 or やや上げる。
- **rejected** で「単価が合わない」が多い → 単価の Hard Filter または単価フィットの重みを確認する。
- **rejected** で「スキル不足」が多い → 必須スキルの重みを上げる、または Hard Filter を厳しくする。

定期的（月1回等）に outcome と outcome_reason を集計し、CONFIG の重みや閾値を調整する。

---

## 5. 実装時の注意

- outcome の更新は **人が行う**（自動で accepted/rejected にしない）。
- outcome_at / outcome_by は監査と分析のため必ず保存する。
- 個人情報を含めない形で **集計レポート** を出せるようにすると、評価ループが回しやすい。

以上を「評価ループ」の仕様とする。
