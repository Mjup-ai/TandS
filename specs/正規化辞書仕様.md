# 正規化辞書仕様

検索精度＝使いやすさのため、スキル・駅・単価・雇用形態を正規化する。  
「JavaでJavaScriptが当たる」問題を二度と起こさない。

---

## 1. スキル辞書（ID化必須）

### ルール

- **JAVA** と **JAVASCRIPT** は別IDで管理する。
- **React.js / ReactJS / React** は同一スキルIDに寄せる（`skill_aliases` で同義語を登録）。
- 検索の基本は**スキルID一致**。部分一致・あいまい検索は「ゆる検索」として別機能で提供する。

### テーブル

- **skills**: `id`, `code`（正規化コード・一意）, `name`, `category`
- **skill_aliases**: `skill_id`, `alias`（表記揺れの文字列）

### 運用

- 初版は人材メール10〜20通から出現したスキル名を洗い出し、同義語を登録して辞書を作成する。
- 新規スキルが出た場合は「未登録」としてフラグを立て、辞書に追加後に再正規化できるようにする。

---

## 2. 駅（最寄駅・勤務地）

- 表記揺れを統一する（例：新宿 / 新宿駅 → 同一 `station.code`）。
- **stations**: `id`, `code`, `name`
- 地域・路線でグルーピングする場合は `category` や親IDを拡張で追加可能。

---

## 3. 単価

- 70万、700,000、スキル見合い などの表記を正規化する。
- 単位は「万円」に統一。
- 不明・曖昧な場合は null とし、**推測で埋めない**。confidence を低くして「要確認」に回す。

---

## 4. 雇用形態

- 正社員 / 契約 / 業務委託 / 個人事業主 等の辞書を用意する。
- **employment_types**: `id`, `code`, `name`
- メール文言から `code` にマッピングするルールを定義する（例：「正社」「正社員」→ 同一 code）。

---

## 5. 正規化フロー（Canonicalize）

1. 抽出結果（固定JSON）の各項目について、辞書を参照してID/正規化値に変換する。
2. マッピングできない場合は「未登録」フラグを立て、confidence を下げる。
3. 根拠行はそのまま保持し、表示時に「抽出元の文言」と「正規化後の値」の両方を出せるようにする。

以上を正規化辞書の仕様とする。
